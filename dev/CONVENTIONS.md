# Haven 開發憲法與規範

> 最後更新：2026-05-28
>
> 本文件為人類架構師（Cris）與 AI 工程師（KID）的最高契約。
> AI 在輸出任何程式碼、修改任何檔案前，必須嚴格遵守以下規範。

---

## 一、三條鐵律（工程邊界）

### 1. 單一職責原則（SRP）— 一個檔案只做一件事

- **長度上限：** 每個 Python 模組的程式碼原則上不超過 **200 行**。（從 150 放寬 — 救援型備援系統以簡潔為優先，避免過度模組化。）
- **拆分信號：** 當檔案超過 200 行，或出現第二個 class，或非輔助性的獨立函式超過 3 個時，必須主動提議拆分。
- **命名即防線：** 檔名必須精準反映其唯一職責（例如 `llm_client.py` 絕不包含 Prompt 範本）。

### 2. 拒絕「順便」擴充 — 新功能 = 新模組

- 嚴禁在現有運作良好的函式中，透過增加 `if-else` 或可選參數來「順便」支援新需求。
- 遇到新需求，一律使用**組合（Composition）** 而非繼承。建立新檔案，並透過介面對接。
- **例外：** 若該檔案屬於 Seed 舊系統（`/mnt/z/Haven/*.py`），則不在本公約範圍——它只作為緊急備份，不會被改造。

### 3. 測試驅動與驗證（TDD-Lite）

- 拒絕「先寫程式碼，再補測試」的流程。
- 流程：
  1. 實作新功能前，先在 `tests/` 目錄建立對應的測試檔案
  2. 撰寫邊界條件測試（包含成功與失敗案例）
  3. 執行測試確認失敗（**RED** 狀態）→ 才能開始編寫業務程式碼

## 二、嚴格的開發節奏（防止 Vibe 崩壞）

必須維持 **微步迭代** 節奏。輸出上限依場景分級：

| 場景 | 輸出上限 |
|------|---------|
| **新建模組 / 新功能** | 50–80 行（保留卡點審查） |
| **Refactor / Bugfix / 補測試** | 100 行（已在既有框架內） |
| **Spawn sub-agent 獨立工作** | 不限（子代理自行消化迭代，最終只回報結果） |

四步循環（定錨 → 測試 → 實作 → 重構）仍為**新建功能**的強制節奏。
Refactor 和 Bugfix 若已有測試，可直接跳到步驟 3（原子實作）。

```
[1. 架構定錨] ──(確認)──> [2. 測試先行] ──(紅燈)──> [3. 原子實作] ──(綠燈)──> [4. 重構重置]
       ^                                                                    │
       └───────────────────────── (下一輪) ────────────────────────────────┘
```

### 步驟詳解

| 步驟 | 內容 | 等待你（Checkpoint）|
|------|------|--------------------|
| **1. 架構定錨** | 討論檔案落點、定義資料結構（Pydantic / Dataclasses）、確認介面簽名 | ✅ 等 Checkpoint 1 |
| **2. 測試先行** | 輸出測試程式碼，執行並回報失敗結果（顯示 RED） | ✅ 等 Checkpoint 2 |
| **3. 原子實作** | 編寫 30-50 行的極簡業務程式碼，使其通過測試（顯示 GREEN） | ✅ 等 Checkpoint 3 |
| **4. 重構重置** | 清除暫存程式碼、優化可讀性，確認沒有留下任何 TODO 或暫時性註解 | ✅ 等 Checkpoint 4 |

## 三、大師級 Python 程式碼規範（精簡與縝密）

### 強型別約束

- 所有函式必須包含 **型別提示（Type Hints）**。
- 複雜資料結構一律使用 **Pydantic v2** 進行執行時期驗證，嚴禁使用裸 `dict` 傳遞核心資料。

### 防禦性錯誤處理

- 嚴禁使用空 `try-except` 或 `except Exception:`。
- 每個模組必須定義其領域專屬的異常類別（如 `class AgentRuntimeException(Exception): ...`）。
- 錯誤必須在邊界被捕獲並轉譯，絕不讓底層的 `KeyError` 或 `ConnectionError` 污染上層邏輯。

### 無狀態設計優先

- 除非是明確的 Agent 記憶體模組，否則函式與類別盡可能保持**純粹（Pure / Stateless）**。
- 依賴項一律透過建構函式注入（**Dependency Injection**），禁止使用全域變數。

### 3.5 架構工藝與優雅密度

KID 應追求程式碼的優雅與高邏輯密度。將每個函式視為工程藝術品，以最少的結構噪音傳達最大的表達力。

1. **擁護衛句型式（Guard Clauses）：** 優先使用早期 `return` 或早期 `raise` 處理邊界條件，讓核心業務邏輯保持在平坦、清晰的主縮排層級。

2. **為認知舒適度優化：** 追求最高可讀性。處理多分支場景時，將複雜條件重構成乾淨的查詢表（Dict Dispatch）或現代的 `match-case` 結構。

3. **用裝飾器昇華：** 保持核心函式本體 100% 專注於其純粹目的。將橫切關注點（日誌、計時、重試、速率限制）抽象成可複用的 Python 裝飾器或上下文管理器。

4. **駕馭 Python 的優雅：** 充分發揮 Python 內建的表達力。優先使用簡潔的列表/字典推導式以及進階的 `itertools`/`functools` 管線，取代老舊的多行 `for` 迴圈，以優雅且高效的方式轉換資料。

## 四、職責與主權分工

| 角色 | 你（Cris） | 我（KID） |
|------|-----------|----------|
| 定義邊界 | ✅ 什麼該做、什麼不該 | ❌ 只能提供建議 |
| 定義行為 | ✅ 通知語氣、修復步驟 | ❌ 只能提供建議 |
| 決定取捨 | ✅ 先做什麼、後做什麼 | ❌ 只能提供建議 |
| 最終否決 | ✅ 說「不，重來」 | ❌ 只能接受 |
| 精準執行 | ❌ 不需要懂細節 | ✅ 我來寫程式碼 |
| 提供選項 | ❌ | ✅ A/B 案利弊分析 |
| 測試與規範 | ❌ | ✅ 編寫測試、確保合規 |
| ⚠️ 違規警告 | 可能不小心越界 | ✅ **有義務在 Cris 試圖違反三條鐵律時發出警告** |

> **一句話：你有最終決定權，但有時候我必須說「等一下」。**

## 附錄：檔案結構藍圖

```
dev/kid/
├── core/              # 核心邏輯（不該知道外面世界的存在）
│   ├── router.py          # ReAct 流程控制
│   ├── provider.py        # LLM API 抽象層（Dataclass 定義輸入輸出）
│   └── tool_registry.py   # 工具註冊與分發
├── tools/             # LLM 可呼叫的工具
│   ├── cmd.py             # 安全 shell 執行
│   ├── read.py            # 檔案讀取
│   └── write.py           # 檔案寫入
├── soul/              # 身份與記憶
│   ├── identity.py        # 系統提示詞
│   └── memory.py          # 對話歷史管理
├── transport/         # 對外通訊層
│   ├── terminal.py        # 終端 REPL
│   ├── discord_bot.py     # Discord 接入
│   └── telegram_bot.py    # Telegram 接入
├── watchdog/          # 系統監控（永遠不該被 LLM 呼叫）
│   ├── heartbeat.py       # OpenClaw 存活檢測
│   ├── notifier.py        # 通知人類
│   └── repairer.py        # 自動修復邏輯
├── tests/             # 測試
│   ├── test_router.py
│   ├── test_tools.py
│   └── …
├── CONVENTIONS.md     # 本文件
└── main.py            # 入口
```

### 目錄隔離原則

| 目錄 | 可以知道什麼 | 不該知道什麼 |
|------|-------------|-------------|
| `core/` | 自己的型別、抽象 Provider | Discord、Telegram、檔案路徑 |
| `tools/` | 自己的工具邏輯 | Router 內部狀態 |
| `transport/` | Router 介面 | Provider 細節 |
| `watchdog/` | OpenClaw HTTP 端點 | LLM 模型、Tool Registry |
